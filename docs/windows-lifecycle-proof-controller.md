# Windows lifecycle proof controller

The installed collector-service acceptance matrix must run through one source-controlled controller. Temporary PowerShell orchestration, ad hoc service commands, and repeated elevation attempts are not evidence.

This document is the implementation gate for the attended #69/#70 Windows proof. A private source-controlled controller now owns the fixed plan, exact artifact binding, authenticated one-elevation broker, protected evidence root, and Job Object process settlement. Its mutation entry remains deliberately fail-closed with `lifecycle_controller_not_reviewed` until the complete lifecycle sequence and independent review are accepted. No installed lifecycle evidence is claimed yet.

## Fixed architecture

The controller is a private Rust proof binary with two fixed modes:

1. A standard-user parent performs every read-only preflight available to its token, creates a 256-bit nonce and independent monotonic send/receive sequences, retains no-write/no-delete handles for the exact installers, launches standard-token desktop phases, and requests one elevation.
2. One elevated worker authenticates the named-pipe peer by PID, process creation time, user SID, session, token elevation, controller file identity, and controller SHA-256. Each nonce-bound message also carries a canonical payload digest and its direction's independent monotonic sequence, so a bounded parent abort can cross a worker checkpoint without sequence collision. The worker creates a protected private staging root and reports its volume-serial/file-index identity; the parent opens and retains its own no-delete handle to that exact directory before accepting any phase or result. The worker completes the privileged pre-mutation observations that the standard token cannot read, copies and revalidates the exact installer bytes, and executes a compiled lifecycle state machine. The request cannot supply a command, arguments, destination, observation, result, or evidence payload.
3. The worker reports `ResultReady` with the exact ordered 28-receipt private manifest while it still owns the protected root and waits. The parent opens and retains every receipt-bound file, derives and validates the sanitized packet from those raw handles plus parent-only desktop/current-user authority, and revalidates every source before sending `EvidenceAccepted`. That acknowledgement releases the worker; it is not publication success. Only after worker exit and outer Job-zero settlement does the parent revalidate every source again, create-new and flush the fixed public export, verify its retained handle/path/identity/bytes/digest, revalidate the sources a third time, and report a passing receipt.

Protocol schema v4 makes each completed lifecycle stage cross an authenticated checkpoint carrying the protected root identity. A parent-side protocol, desktop, artifact, evidence, receipt, timeout, or disconnect failure sends a typed `Abort` when the pipe remains writable and repeats the same authenticated reason if a valid in-flight checkpoint, desktop request, or original `ResultReady` crosses it. The worker stops at the next settled checkpoint, records the typed reason plus its last parent-acknowledged checkpoint and protected-root identity, writes a distinct parent-abort packet, and reports typed restoration truth before waiting for `EvidenceAccepted`. If the worker had already reported a failure, the abort-bound successor preserves that original failure kind, stage, cause, evidence, settlement, and restoration exactly; the parent records its later follow-up error separately. For an original `ProcessSettlement` result, the parent terminates and settles the worker Job before validating the retained diagnostic receipt and does not request new restoration work. Abort evidence handles remain retained through acknowledgement and worker settlement, with the plan artifacts revalidated immediately before acknowledgement and again after settlement. Once the parent may have sent `EvidenceAccepted`, the protocol is terminal: later exit, publication, revalidation, or cleanup errors settle locally and never send `Abort`. If an earlier abort exchange cannot be authenticated, the parent waits boundedly for worker self-settlement before using Job termination as a last resort; that path reports only blocked-untrusted or blocked-unsettled truth unless an authenticated original worker failure is already authoritative.

The thin `scripts/run-windows-lifecycle-proof.ps1` wrapper may hash-check and stage the two plan-bound installers, build the exact controller head, copy those controller bytes out of Cargo's hard-linked output into the private proof staging root, and select `preflight` or the explicit `-Run` entry. It must not own installer arguments, service actions, process settlement, cleanup, observations, or evidence decisions.

Every installer, uninstaller, desktop, WebView, and proof child belongs to an owned kill-on-close Job Object with an absolute deadline. The worker creates each child suspended, assigns it to the Job, resumes it, waits for the entire process tree, and revalidates the owned bytes only after the Job reports zero active processes. Timeout terminates the Job and proves settlement before any cleanup or further mutation.

The elevated broker starts from the canonical system directory instead of the invoking user's working directory. Installer and uninstaller trees receive a fixed Unicode environment containing only canonical machine `ComSpec`, `Path`, `SystemDrive`, `SystemRoot`, and `WINDIR` values plus `TEMP` and `TMP` bound to the protected evidence root. `SystemDrive` is derived from the canonical Windows directory rather than inherited from the caller; this keeps native property-system cache resolution absolute without widening the environment. Their current directory is also the protected evidence root. The controller does not inherit caller-controlled process lookup or extraction paths into a privileged mutation.

The elevated worker must run a protected copied uninstaller with:

```text
/S _?=C:\Program Files\BatCave Monitor
```

`_?=` is last and unquoted. Waiting on the installed `uninstall.exe /S` launcher observes only NSIS's outer self-copy stub and can report a false zero while the real uninstall fails. See the [NSIS command-line reference](https://nsis.sourceforge.io/Docs/Chapter3.html) and [uninstaller error-level behavior](https://nsis.sourceforge.io/Docs/AppendixD.html).

## Preflight gate before UAC

The parent stops before elevation unless all of these checks pass:

- The repository, branch, exact commit, installer paths, file sizes, and SHA-256 values match the requested proof packet.
- The controller and both worker modes passed unit tests and exact-head build validation.
- The current machine state is one documented allowlisted starting state; an access error or malformed state is not treated as absence.
- The standard token captures the install root, service status, uninstall registration, current binaries, and product processes. The parent rejects thread impersonation, retains its explicit primary token, and binds the token SID, session, and logon LUID to the explicit-token Profile and `FOLDERID_LocalAppData` directory identities. The process `LOCALAPPDATA` directory must reopen to that same identity. Profile, LocalAppData, and the resolved BatCave root are verified component by component without following reparse points, retained with delete-sharing, and freshly reopened during revalidation. The elevated packet still reports current-user state as `Unknown`; elevation never derives that path from its environment. The authenticated elevated worker captures the service DACL and protected service data root before its first mutation; neither ACL is weakened for proof convenience.
- No BatCave desktop, WebView, installer, uninstaller, or prior controller process is running.
- The fixed protected `%ProgramData%` evidence leaf for the nonce does not already exist. The controller never reuses or clears a stale evidence root.
- Every component of the fixed `artifacts/windows-lifecycle-proof` destination ancestry is opened without following reparse points and retained without delete sharing before elevation. The fixed `windows-lifecycle-proof.sanitized.json` leaf must be absent; the controller never removes or overwrites a stale export.
- An independent review accepted the exact controller head.

Every environmental probe returns exactly one of:

```text
Present(value) | Absent | Unknown(error)
```

Only documented not-found results are `Absent`. Access denial, malformed registry or SCM data, WMI/CIM failure, ETW query failure, pipe-query failure, and timeout are `Unknown` and stop the proof.

## Closed lifecycle sequence

The worker owns the sequence and does not improvise after failure:

1. Capture the complete initial state.
2. If the machine is in the allowlisted stopped `1066/1` legacy state, run the exact final candidate as `/S /UPDATE` so the new staged transaction repairs it.
3. Prove the exact final service process identity, path and digest, pipe server PID, production-shaped active ETW lease and observed session identities, lock ownership, zero loss counters, clean failure-marker state, standard-token desktop fallback/privileged state, and process-tree settlement.
4. Run the protected copied final uninstaller with the direct `_?=` form and prove total product absence.
5. Install the exact baseline.
6. Exercise clean stop/start, second desktop instance, crash/retained ownership, restart/recovery, and standard-token desktop states.
7. Seed the exact historical CLI bytes and known retired-helper artifacts plus an unknown helper sentinel.
8. Upgrade to the exact final installer.
9. Prove the final service generation and that the historical CLI and known helper artifacts were removed while the unknown sentinel was preserved.
10. Exercise final clean stop/start, crash/recovery, and incompatible/missing/stopped fallback states.
11. Uninstall through the protected copied uninstaller and prove final residue and current-user retention policy.

Reboot and independent multi-user evidence remain separate attended packets when they cannot be completed inside the same worker lifetime.

## Required observations

Each stage records bounded private JSON with exact machine paths under the protected local evidence root. A distinct sanitized export replaces those paths with logical root identities and relative leaves before it can leave the machine.

Each stage records:

- service SCM state, PID, creation time, exact image path and digest, account, type, start mode, exit codes, recovery configuration, owner marker, and service DACL;
- pipe presence and server PID;
- production ETW lease identity, independently observed session identity, controller PID plus creation time, exact service-instance identity, loss counters, and owner/process locks;
- stable, staged, rollback, journal, atomic temporary, failure-marker, monitor, CLI, uninstaller, and install-root identities;
- Add/Remove Programs key, machine product key, HKCU autostart value, public desktop shortcut, common Start Menu shortcut and target;
- standard desktop and WebView process tree, token elevation, session, executable identity, and visible collector status;

Final absence requires no service or service registry key, pipe, ETW session, lease/locks, installer-owned Program Files leaf or directory, machine registration, shortcut, autostart value, or BatCave process tree. Declared current-user state is preserved unless the proof explicitly selects its separate deletion policy.

## Failure behavior

Any failed mutation stops after one fixed attempt. When the owned Job settles, the worker writes one stage-bound private packet containing the last settled machine snapshot, the terminal process-tree state captured before termination, the post-settlement machine snapshot, and executable revalidation truth. The private forced-crash primitive likewise records whether the exact service process, SCM state, and lifecycle marker settled. Every `WorkerFailure` carries a structured restoration outcome: not required, restored with a receipt, blocked by unsettled state, blocked because the state is no longer trusted, or failed with its own reason and receipt/write error. `BlockedUnsettled` is valid only when the overall worker, desktop, or service settlement remains unproven; the failure kind continues to identify the primary cause. It is the only validated outcome that makes the parent terminate the worker without sending `EvidenceAccepted`. Mutation failures always require a restoration disposition. Evidence-write and controller failures may use `NotRequired` only before any mutation; after a completed mutation stage they must report restored, blocked-untrusted, failed, or overall blocked-unsettled restoration truth. A restoration receipt is distinct from the failure receipt and uses the fixed `<stage>-restoration.private.json` leaf for the attempted or last completed mutation stage. When process or service settlement cannot be proven, the worker writes only a clearly marked pre-settlement diagnostic packet and reports restoration as blocked; the parent then terminates and settles the outer worker Job before opening any failure receipt. Evidence-write and process-settlement failures remain distinct. For all other settled failures, the parent verifies the fixed stage-bound failure and restoration receipts while the worker waits for `EvidenceAccepted`, retains the verified file handles, acknowledges, then proves worker settlement and revalidates the plan-bound artifacts. A later parent settlement, artifact-revalidation, or receipt-verification failure is appended to the original structured worker failure instead of replacing its stage, cause, or receipt. The controller does not retry with a different command, controller, installer path, uninstaller form, service tool, or cleanup strategy.

One fixed restoration path may run only after the failed mutator and its owned process tree have settled and the resulting machine state remains trusted. Unproven settlement blocks further mutation. Restoration must preserve the original stage failure and report its own verified result separately; it is not permission to choose another command or cleanup strategy.

The stage-specific restoration matrix does not introduce a second installer, service, or cleanup engine. After any settled failure, the worker captures a fresh elevated snapshot and validates the exact last completed stage. A mutation failure writes the fixed restoration leaf for its attempted stage and names the last completed stage as its target; a checkpoint, controller, evidence, desktop, or parent-abort failure uses the completed stage for both. `Restored` means that this fresh snapshot satisfied the complete target validator and the restoration packet was written successfully. It does not imply that the proof controller ran an additional mutation. The standard parent opens the retained receipt, parses its deny-unknown typed packet, binds its stage, target, method, disposition, and reason to the worker result, revalidates the snapshot, and then revalidates the retained file handle. A target mismatch reports `Failed(lifecycle_stage_restoration_target_invalid)` with the observed packet; an evidence-write failure reports `Failed(lifecycle_stage_restoration_evidence_failed)`. Neither path attempts another mutation.
