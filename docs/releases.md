# Release channels and verification

BatCave publishes immutable GitHub Releases from commits already on protected `main`. The package version in `src/BatCave.App/src-tauri/Cargo.toml` is the only authored app SemVer. Tauri inherits that value for the runtime and platform bundles; the private npm package and lockfile do not carry a second app version. `node scripts/verify-release-version.mjs <tag>` requires the release tag to match Cargo before any platform build starts.

Release tooling keeps syntax parsing separate from repository verification. `parseReleaseTag` is the pure API for table-driven contracts and tests that use synthetic tags. Every executable release boundary must call `verifyWorkspaceReleaseVersion` before it generates or verifies artifacts; that API reads the Cargo package version and rejects drift. New release scripts must preserve that split rather than passing a placeholder version into the bound verifier.

Stable tags use `vMAJOR.MINOR.PATCH`. Prerelease tags add a SemVer suffix such as `v0.2.0-rc.1` and are always marked as GitHub prereleases. Stable and prerelease artifacts never share a tag or installed version.

The `Versioned release` workflow runs only through a manual dispatch from `main`. Enter the tag explicitly, select its stable or prerelease channel, and supply the exact 40-character source commit SHA approved for release. The workflow has no reusable tag default. It requires that SHA to be both the checked-out commit and the current tip of protected `main`; it does not publish from a pushed tag. `publish: false` is the default dry run and retains the complete workflow artifact without creating a tag or GitHub Release. Set `publish: true` only for an approved release.

All build and publication jobs enter the protected `release` environment. Configure its `RELEASE_ADMIN_READ_TOKEN` secret with a fine-grained personal access token or GitHub App token that can read repository Administration settings. Each sensitive job uses that credential only for its first control check, before reading signing secrets or changing release state. The workflow's ordinary `GITHUB_TOKEN` remains limited to the repository permissions needed by later artifact and release operations.

Every release artifact contains the offline-capable Windows NSIS installer, Windows GUI and benchmark CLI executables, Linux deb and AppImage packages, the universal macOS DMG and updater archive, `SHA256SUMS.txt`, and a Sigstore/GitHub build-provenance bundle. Workflow artifacts are retained for 30 days; published GitHub Release assets are durable.

Successful `main` pushes seed the Rust dependency caches that pull-request validation restores. Pull requests remain restore-only and cannot publish Rust caches. The existing `Platform bundles` jobs build each Windows, Linux, and macOS release graph once, retain the platform artifacts, and seed the matching dependency-only release cache; validation does not run a second Linux bundle solely for caching. Versioned tag and manual release jobs restore those trusted dependency caches but never save tag-scoped or workspace-crate output. Every workflow action is pinned to an immutable commit, and the cache policy changes dependency reuse only without removing a validation, packaging, signing, notarization, or provenance gate.

The independent Linux package-transport validation job restores the release-profile dependency cache written by the trusted `Platform bundles` workflow but never saves it. After building the deb and AppImage, the job runs the ignored package-transport tests with Cargo's release profile so dependency artifacts can be reused without caching workspace crates. This changes build reuse only; all package and transport gates remain required.

On Windows, `batcave-monitor-cli.exe` is a standalone release asset for benchmarks and automation; it is not an NSIS-installed product component. The per-machine installer owns the desktop and collector service binaries only. Upgrade and uninstall recognize the single historical CLI payload by its exact size and SHA-256, remove those bytes through the verified open file handle, and reject any different object at that fixed path as untrusted residue.

The Windows installer no longer creates Public Desktop or Common Programs shortcuts. Install, update, repair, rollback, and uninstall keep both shared paths absent. BatCave's pinned Tauri 2.11.4 NSIS template makes both generated shortcut callbacks return before any legacy-target inspection or mutation, removes the finish-page desktop-shortcut control, and suppresses Tauri's stock path-based uninstall cleanup after native PREUNINSTALL retirement while preserving its AppUserModelId cleanup. Windows bundle validation and the protected release build verify that contract against the exact generated release `installer.nsi`. Exact historical Tauri links are retired through a no-follow, single-link, handle-validated native verb bound to a verified stable, recovery, or semver-staged controller. OS-known folders and the controller's pinned Program Files install root supply the paths; verbatim UNC handle paths are normalized to their canonical redirected-known-folder form and no `C:` layout is assumed. Unknown objects fail closed and no shared Start/Desktop ACL is changed. Upgrade preflight runs with the prior verified service active, and every final absence failure rolls an in-flight candidate back before the upgrade journal can be finalized; a fresh-service failure uses the new-install rollback. There is intentionally no installer-created per-user replacement in this slice because a per-machine Apps & Features uninstall cannot authenticate and retire links for every user profile, especially with over-the-shoulder UAC credentials, logged-out users, or redirected Start folders. The [shared-shortcut decision](decisions/0013-windows-shared-shortcut-retirement.md) records the launch and proof boundary.

Lifecycle evidence requires both exact historical links through the baseline and seeded-legacy stages, then requires both absent from final repair onward. Initial and final uninstall require total absence, and unknown observations fail closed.

## Windows collector privilege migration

The collector-service cutover removes the launch-time elevated-helper mode, its `set_admin_mode` command and persisted preference, helper IPC and snapshot contracts, and the `elevated_helper` runtime source. Protected Windows collection now comes only from the authenticated installed collector service or an already-elevated desktop process. Missing, stopped, incompatible, or unauthorized service states remain visible and fall back to standard-access collection.

Upgrade startup performs bounded migration cleanup for historically shipped helper artifacts under the current user's `BatCaveMonitor\elevated-helper` directory. It removes only the known files and valid run-directory shapes, preserves unknown or unsafe entries, and never follows reparse points or recursively deletes an untrusted tree. Source validation and packaging prove the retired path is absent from the new build; installed upgrade, restart, multi-user, rollback, and uninstall behavior still require native Windows lifecycle evidence before release acceptance.

Collector-service upgrades use a fixed recovery-controller copy of the new service plus a protected, atomically written transaction journal. Preparation also creates a verified installed-version compatibility alias. That alias becomes an uninstaller recovery path after an uninstaller containing this lookup has shipped; the first migration from an older uninstaller remains installer-retry-only if it fails before replacing `uninstall.exe`. Before the stable service image is replaced, the controller records exact old/new digests and identity-bound artifact names and creates the digest-named rollback image through a synced atomic replacement. After replacement it accepts the candidate only when SCM reports `Running`, the process is the generation started for the transaction, the executable resolves to the exact stable path and new digest, and the service has produced an initial telemetry sample; otherwise it restores and starts the verified old image. Interrupted preparation, candidate validation, rollback, and finalization are resumable, including same-version different-build retries, missing-stable rollback, and recovery by a superseding installer. Uninstall can remove a settled stopped `1066/1` service without restarting the broken old image, preserves the journal until SCM deletion succeeds, and then retires exact installer-owned upgrade residue; delete-pending cleanup reports that a reboot is required instead of claiming completion. These are source and package guarantees only until the exact built installers pass the native lifecycle proof matrix.

The private Windows lifecycle proof controller now binds a fixed baseline/final installer plan to owned file handles, authenticates one elevated worker over a local protected named pipe, and supervises installer descendants with a kill-on-close Job Object plus absolute settlement deadlines. Privileged installer trees now start from the protected evidence root with a fixed machine-derived Unicode environment instead of inheriting caller-controlled lookup, temporary, or working-directory state. Failed mutations now preserve stage-bound pre-settlement process truth and, when settlement succeeds, a post-settlement machine snapshot with a structured restoration outcome. The worker also retains the exact historical CLI before repair, drives baseline stop/start through the production provisioner, and owns an exact-service-hash-verified forced-crash primitive whose failure remains unsettled until SCM, the process handle, and the lifecycle marker prove release. Retained service fixtures are independently pinned by source commit, size, SHA-256, behavior, ProductVersion, and fixed artifact path for the real transport-verified incompatible-service phase and the fixed SCM-start-failure rollback phase. The rollback phase now feeds plan-bound bytes into a private feature-gated adapter around the existing production prepare, candidate commit, rollback, and transaction cleanup machinery; it adds no production verb or second state machine, and proof success requires the exact original digest to remain running after all journal-bound residue is gone. Every success leaf has one explicit deny-unknown private-envelope binding with receipt-bound machine, desktop, crash, or rollback payloads. The writer validates and size-bounds the exact pretty-JSON bytes that it then persists and receipts. Raw capture records exact service-process creation time, typed service/root ACL observations and their exact DACL digests, excludes only the authenticated parent and worker controller generations, attributes WebView residue only through a bounded WebView-only ancestry chain ending at the exact fixed installed monitor image path, treats an unreadable candidate anchor as `Unknown`, includes the historical CLI in residue scans, and drops source handles before uninstall or installed-image mutation boundaries. Raw machine packets now also require the fully verified service-pipe PID/start, the protected bounded-parsed ETW lease, native ETW identity and loss statistics, separate ETW-owner and service-lifecycle lock observations, and elevated service/install residue authority. That residue authority covers the fixed service registry key with bounded `BatCaveLastFailure`, the ServiceData upgrade journal and owned atomic leaves, and the Install staged, digest-named rollback, rollback-temp/atomic, and feature-gated rollback execution-marker leaves. Direct-child sets and matching files are bounded, pinned, DACL/no-follow verified, enumerated twice, and become `Unknown` on churn; capture handles are dropped before mutations. The unchanged sanitized v2 `failure_marker` represents only the Install rollback execution marker, the journal is rooted at `ServiceData`, and atomic files may be rooted at `ServiceData` or `Install`. Elevated capture now also verifies the exact 64-bit product-registration key and unnamed Install-root `REG_SZ`, requires the 32-bit view to be absent, and rejects named values, subkeys, key-path, last-write, enumeration, or DACL drift. It resolves and pins the Public Desktop and Common Programs ancestry and parses each fixed shortcut through native `IShellLinkW`/`IPropertyStore` without `Resolve` while its no-write/no-delete-sharing leaf handle remains pinned. Exact target, arguments, icon, working directory, show command, hotkey, description, and AppUserModelID are required; file identity, digest, DACL, known-folder results, and collisions with the other shortcut or installed binaries are rechecked. Public Desktop is explicitly a recaptured point-in-time observation and makes no persistent tamper-resistance or unprivileged-write-denial claim; Common Programs retains the stricter no-untrusted-writer policy. The parent accepts projection input only as receipt-bound verified file handles from the protected evidence root, independently enforces the exact private/sanitized manifest, parses the 28 typed private leaves, and derives those raw-backed runtime, residue, product-key, shortcut, HKCU autostart, helper-residue, and sentinel fields before comparing them with the worker packet and six live parent desktop results. Duplicate or reused receipts, swapped payloads, changed bytes, sanitized drift, parent drift, private path leakage, identity mismatch, loss-counter overflow, or unknown raw state fails closed. Projection still ends with `lifecycle_private_projection_raw_authority_incomplete` pending protocol v4 parent derivation and export authority. For success and settled failures, the worker reports the protected evidence-root file identity and waits at `ResultReady`; the parent opens and revalidates every reported receipt while the worker is still alive, sends `EvidenceAccepted`, and only then proves worker settlement. A validated `BlockedUnsettled` result is the fail-closed exception: the parent terminates and settles the worker without acknowledgement before opening its diagnostic receipt. The mutation entry and both readiness gates remain disabled until protocol v4 parent-owned export, the general restoration matrix, installed UI Automation smoke, and independent review are complete, so this source addition does not yet satisfy the installed #69/#70 proof gate.

The private worker stage engine now reaches `FinalUninstall` and writes the exact 28 protected private leaves, including all six authenticated desktop phases and provisioner-owned missing/stopped/incompatible service transactions. Those transactions attempt exact final-running restoration only after the body proves settlement; an unsettled desktop or body result blocks further mutation and remains `BlockedUnsettled`. Protocol v3 still stops deliberately at the exact settled controller failure `lifecycle_parent_export_pending`; that failed result carries no success receipts or sanitized export, so the 28 files are not parent-accepted proof. Both readiness gates remain false pending protocol v4 parent derivation and durable export, the general restoration matrix, exact installed UI Automation evidence, and independent final review.

Desktop phase transport now allows eight minutes for each complete bounded UI/process phase instead of racing the phase's individual 30-second operations. Missing authenticated desktop completion remains `BlockedUnsettled`, skips fallback restoration, and blocks current-user residue cleanup unless the standard parent separately proves every parent-owned desktop Job settled. Failed desktop results, fallback attempted stages, crossed pre-acknowledgement messages, and the terminal parent-export checkpoint are now bound exactly before acknowledgement.

The standard parent now rejects thread impersonation, retains its explicit primary token, and binds the token SID, session, logon LUID, profile, `FOLDERID_LocalAppData`, and resolved current-user data root. Profile, LocalAppData, and data-root components are opened without following reparse points; token-derived LocalAppData must match the live process environment by directory identity. At the authenticated `FinalFallbackStates` and `FinalUninstall` checkpoints, with acknowledgement withheld, the parent captures and revalidates `settings.json`, `warm-cache.json`, and `diagnostics.jsonl` through handles that allow writes, renames, and deletes. Projection accepts only exact before/after identity, size, and digest equality, and exposes that narrow result as `declared_current_user_objects_preserved`; it does not claim whole-root preservation or cover rotated `diagnostics.jsonl.1`. The worker now reaches both final-stage anchors, but protocol v3 deliberately returns `lifecycle_parent_export_pending` without success receipts, so neither readiness gate is enabled.

The standard parent now also owns the raw current-user residue timeline, including an independent clean `InitialState` capture and projection rather than an alias to `FinalRepair`. It observes only the shared HKCU view and requires `Run\BatCave Monitor` to be absent or the exact quoted fixed-monitor `REG_SZ`, with native key path, DACL writer policy, bounded enumeration, last-write, token, profile, LocalAppData, and data-root revalidation. The Run-key owner may be only the parent SID, LocalSystem, or Builtin Administrators and remains anchored across captures; helper directories and files remain parent-user-owned. With the `BaselineRollbackRecovery` acknowledgement withheld, the parent journals the prior absent Run value plus preexisting helper-directory identities, ACLs, and manifests before transactionally creating the exact Run value, eight fixed helper fixtures, and one unknown sentinel. Any partial seed failure rolls back only recorded exact objects. The bounded helper observer pins the fixed root and proof-run directory without following reparse points; records identity, digest, owner, DACL, unexpected count, and manifest digest; rejects hardlinks, escape, overflow, or churn; and drops every capture handle before acknowledgement or desktop launch. Stage projection requires HKCU absent through baseline, exact from legacy seeding through fallback restoration, and absent after final uninstall. Known helper files remain through the pre-launch missing-service state, are absent when that first post-upgrade standard desktop launch completes, and the sentinel remains identity/hash stable through final-uninstall evidence validation. After success, worker failure, or abort, parent cleanup runs only after the worker and parent-owned desktop trees are proven settled. It deletes only unchanged no-follow single-link transaction files, the exact controller Run value, and empty identity-matching controller-created directories; changed or unexpected data is preserved and reported as cleanup-blocked, while unproven settlement leaves residue in place and reports `BlockedUnsettled`. None of the SID, native HKCU key path, or absolute LocalAppData state enters the request, worker, private packet, or unchanged sanitized v2 schema; protocol v3 and the fixed 28 leaves are unchanged. The worker reaches the final checkpoint; the parent-owned protocol v4 export and general restoration work remain incomplete, and both readiness flags remain false.

Authenticated root-bound stage checkpoints and a typed abort exchange prevent parent-side protocol, desktop, receipt, evidence, timeout, and disconnect failures from immediately dropping the worker Job. The parent repeats the same abort when a valid in-flight checkpoint, desktop request, or original result crosses it; the worker records its last acknowledged checkpoint, writes abort and restoration evidence, and waits for parent receipt validation. A late abort cannot replace an existing worker failure: the successor preserves its kind, stage, cause, evidence, settlement, and restoration, while the parent appends receipt, artifact, acknowledgement, or settlement errors separately. An original `ProcessSettlement` receipt retains `BlockedUnsettled` truth and is opened only after the parent settles the Job. Abort receipt handles remain retained through acknowledgement and settlement, and plan artifacts are revalidated before acknowledgement and after settlement. If the abort exchange fails, bounded worker self-settlement precedes last-resort Job termination; an authenticated original worker failure remains authoritative, while a resultless path reports blocked-untrusted or blocked-unsettled truth only. The source-controlled stage restoration matrix now validates a fresh elevated snapshot against the exact last completed target and writes an attempted-stage-bound receipt; it claims `Restored` only for a matching target plus durable evidence and never launches a second mutator. Target or evidence failure remains typed, and an unsettled process or service still forbids further work. The existing fallback closures remain the only provisioner-owned restoration transactions. Both readiness gates remain false pending independent review, native installed UI Automation, and exact-artifact proof.

The private feature now also contains the standard-token desktop runner. Desktop launch failures carry exact settlement state; the child environment is a fixed allowlist derived from the authenticated standard-token profile and fixed Windows paths; Job membership is enumerated directly until a bounded stable process set is retained by process and image handles; and production service-peer verification accepts only the explicitly retained installed desktop and service identities without weakening the normal `batcave-monitor.exe` policy. Pinned `uiautomation` selects the unique stateful header Diagnostics button, preserves the resource rail's visible Diagnostics state, and reads visible, enabled rows from the exact modal subtree. Each requested phase revalidates the retained HWND/PID/process generation and retries only missing, no-sample, non-current, connecting/recovering, or other well-formed state/source convergence for at most 30 seconds; ownership, authorization, terminal service state, identity, malformed-contract, and unknown failures stop immediately. Second-instance focus converges on the exact retained primary HWND and process generation. The standard-fallback collector reports that process ETW was constructed disabled; the runtime authenticates that report against its own Windows platform, standard process token, source `none`, and inactive-service state before publication, clamping every mismatched context to false. Runtime/protocol adapters preserve the authenticated value, service and fixture samples are false, and protocol validation independently accepts true only for that same standard-token context. Elevated, local-process, unknown-elevation, non-Windows, and active-service claims fail validation, and Diagnostics maps only validated true authority to `Disabled`. An installed-package UI Automation tree smoke is still required before this runner can count as native proof.

Because the NSIS postinstall hook runs after package files and registration are written, a rejected service candidate can temporarily leave the new desktop package beside the restored old service. That state is recoverable rather than successful: protocol version checks keep the desktop on its visible standard-access fallback, the retained recovery controller supports installer retry, the compatibility alias supports future uninstallers that contain the lookup, and final uninstall verifies that the machine-wide installer-location key is absent without deleting declared current-user state. Native fault injection must still prove this mixed-version recovery path.

Windows proof builds can now export the exact post-sign NSIS uninstaller bytes through an opt-in `BATCAVE_UNINSTALLER_EXPORT_PATH`; a failed export fails the build, avoiding an extra install merely to discover the plan-bound uninstaller identity.

Residue enumeration is capped per root at 256 direct children, 64 classified leaves, and 256 MiB of classified bytes, with tighter per-kind caps. Hardlinks, noncanonical case aliases, and count/byte overflow fail as `Unknown` before an unbounded hash can begin.

## Platform support and proof

The [platform capabilities matrix](platform-capabilities.md) is the canonical human view of supported release profiles; the [version 1 platform support contract](evidence/releases/platform-support-contract.v1.json) is the machine authority. `declared` records the intended host, architecture, runtime, and package boundary. `source_enforced` records that repository configuration, hosted builds, metadata, and extraction-only package checks agree with that boundary. Every current profile still has `native_oldest_supported: pending`, so none of those checks proves installation or runtime behavior on its oldest-supported host.

Linux release builders are pinned to `ubuntu-22.04`. Package verification requires x86-64 ELF payloads, a maximum required symbol version of `GLIBC_2.35`, and deb dependencies on `libgtk-3-0` and `libwebkit2gtk-4.1-0`. Those are source/build gates only. Native evidence and the independent release decision remain separate requirements.

Verify a downloaded file with `Get-FileHash -Algorithm SHA256` on Windows, `sha256sum --check SHA256SUMS.txt` on Linux, or `shasum -a 256 -c SHA256SUMS.txt` on macOS. Verify provenance with `gh attestation verify <file> --repo TheGreenCedar/BatCave`. On Windows, confirm the installed version in Apps settings and the executable file properties matches the release tag without the leading `v`.

After publication, the release workflow downloads every expected asset again through its unauthenticated public URL into a new directory. It rejects any name, size, or SHA-256 difference from the prepublication inventory, verifies that `SHA256SUMS.txt` covers every build subject, requires GitHub's immutable-release attestation, and verifies each subject against the exact `main` source SHA and `.github/workflows/release.yml` on a GitHub-hosted runner. Passing contract tests proves the verifier source; only a successful run against the published assets proves a release, and that live evidence remains part of the stable-release gate.

The [public-artifact install smoke harness](public-artifact-install-smoke.md) consumes the process-local JavaScript verifier receipt for planning and offers fixture-driven contract tests. Its owned-byte receipt remains artifact-only. The private `batcave-install-smoke` Rust binary is a separate production entry: it independently verifies the immutable public release, complete inventory, checksums, source-bound attestations, and selected bytes. On Linux it seals the verified deb or AppImage bytes in a private immutable descriptor and dispatches that authority to the Linux handler. The handler currently revalidates the descriptor and returns `skipped`; it does not install, stage, launch, or emit native or release evidence. The [native executor boundary](native-install-smoke-executor.md) and [Linux adapter boundary](linux-native-install-smoke-adapter.md) record that split.

Published deb and AppImage releases also run separate protected post-public smokes on fresh `ubuntu-22.04` GitHub-hosted runners. Both compare anonymous public bytes to the exact pre-publication candidate inventory, verify checksums and source-bound attestations again, and exercise identity, settings, degradation, advancing core telemetry, removal, and cleanup as a standard user. The [deb job](linux-deb-post-public-smoke.md) installs and purges through owned transient systemd units. The AppImage job additionally verifies the updater signature, then uses fixed extract-and-run staging without claiming a normal package install or A-to-B update. Both sanitized results are deliberately post-public observations with `release_evidence_eligible: false`; neither promotes the underlying current-user `native_candidate` packet into a #98 `release_evidence` packet.

Published universal macOS updater archives run a separate [protected post-public staging observation](macos-updater-post-public-smoke.md) on `macos-15`. The private Rust verifier independently binds the immutable public release, full inventory, checksums, source attestations, updater signature, and exact archive bytes before it preflights, materializes, reverifies, and removes one private staged app tree. The result remains explicitly ineligible as release evidence and preserves `macos_updater_staging_only`; it does not install or launch the app, prove Developer ID/notarization/stapling at the staged destination, exercise A-to-B updating, or alter the unresolved DMG transport boundary.

After the platform lanes produce sanitized packets for one exact public release, assemble `docs/evidence/releases/<tag>/index.json` and run `node scripts/validate-release-evidence-index.mjs` against it. The index binds packet file digests, release/workflow identity, support profiles, package roles, and selected public assets; it deliberately has no passing or accepted disposition. Its successful validation proves only that the review input is internally consistent. The release still requires live public verification, native platform evidence, updater proof, and the independent #76 release decision.

Windows artifacts remain unsigned until the code-signing issue is resolved. Do not promote an unsigned prerelease to the stable channel.

## macOS signing and notarization

Pushes to `main` and manual `Platform bundles` runs produce an ad-hoc-signed universal `.app` and DMG for internal validation. They are not notarized public downloads. Versioned releases require a Developer ID Application certificate and App Store Connect API key; the workflow will fail before packaging if any required secret is absent.

Configure these GitHub Actions secrets:

- `APPLE_CERTIFICATE`: base64-encoded Developer ID Application `.p12` contents.
- `APPLE_CERTIFICATE_PASSWORD`: password used when exporting the `.p12`.
- `APPLE_SIGNING_IDENTITY`: the complete `Developer ID Application: Name (TEAMID)` identity.
- `APPLE_API_KEY`: App Store Connect API key ID.
- `APPLE_API_ISSUER`: App Store Connect issuer ID.
- `APPLE_API_KEY_CONTENT`: the complete private `.p8` file contents, including its BEGIN/END lines.
- `TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`: the existing Tauri updater signing credentials shared by all platforms.

The release job writes the API key and certificate to mode-600 temporary files, imports the certificate into a temporary keychain, and lets Tauri sign, notarize, and staple the universal app. It then signs, notarizes, and staples the containing DMG separately before removing every temporary credential in an `always()` cleanup step. `scripts/verify-macos-bundle.sh --mode release` requires both `arm64` and `x86_64` slices, macOS 12 as the deployment minimum, hardened runtime, one consistent bundle ID and Developer ID team, accepted Gatekeeper assessments, valid app and DMG staples, and a healthy DMG filesystem. The same checks apply to the app mounted from the DMG and the app extracted from the updater archive. The release is blocked if any gate fails.

For a downloaded public DMG, run:

```bash
hdiutil verify BatCave*.dmg
spctl --assess --type open --context context:primary-signature --verbose=4 BatCave*.dmg
xcrun stapler validate BatCave*.dmg
```

Mount the image and run `spctl --assess --type execute --verbose=4` against `BatCave Monitor.app` before first launch when performing release QA on a clean machine.

## Signed in-app updates

BatCave never checks for updates at startup or in the background. The Settings drawer provides a manual **Check now** action that contacts `github.com` with a 15-second timeout. It reads only the latest stable release; prereleases and downgrades are not offered. A failed or offline check leaves monitoring unchanged.

Tauri updater signatures are mandatory and independent from Windows Authenticode or Apple Developer ID signing. Release builds use `TAURI_SIGNING_PRIVATE_KEY` and `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` from GitHub Actions secrets, while the public key is embedded in `tauri.conf.json`. The release workflow creates signed NSIS, AppImage, and universal `.app.tar.gz` updater artifacts plus `latest.json`; missing, empty, duplicate, or byte-mismatched signatures fail verification. The macOS verifier extracts only a private copy of the exact signed archive bytes and rejects absolute or traversing paths, links, device entries, extra roots, and missing or multiple app roots before writing bundle contents. Both `darwin-aarch64` and `darwin-x86_64` manifest entries intentionally point to the same universal archive and signature. Ordinary local builds use no updater signing key and do not create updater artifacts.

The updater signature authenticates payload bytes, not `latest.json`; it does not expire or cryptographically bind a payload to the stable channel. GitHub's latest-release routing, HTTPS, and release controls own that metadata boundary. The exact guarantee and replay limits are recorded in [the updater freshness decision](decisions/0002-update-manifest-freshness-and-expiry.md).

Before replacing a selected update or after download/install completion or failure, the frontend explicitly closes the Tauri `Update` resource. Retry always performs a new check and never reuses the prior in-process selection. The deterministic [local hostile-case matrix](updater-hostile-fixtures.md) exercises the pinned Tauri metadata, comparator, download, and payload-signature boundary without changing production configuration or invoking a platform installer.

Never replace or delete the private key without a rotation release. To rotate, first publish an update signed by the old key that embeds the new public key, then sign later releases with the new private key. If the private key is lost before that transition, existing installations cannot accept another in-app update and users must install a new release manually. Invalid or tampered signatures are rejected by the updater before installation.
